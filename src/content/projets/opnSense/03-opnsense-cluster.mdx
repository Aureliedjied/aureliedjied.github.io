Un pare-feu unique est un point de défaillance critique (SPOF). 
Dans cet article, l'objectif est de préparer l'architecture <strong>Haute Disponibilité</strong> en ajoutant un lien de synchronisation dédié et en déployant un second nœud identique.
</p>

## 🛡️ Le concept de Failover (Basculement)

Le **Failover** garantit la continuité de service. Si le nœud maître (`OPNsense-01`) tombe, le nœud esclave (`OPNsense-02`) prend le relais de manière transparente. Pour y parvenir, les deux pare-feux doivent partager leurs tables d'états via un lien physique dédié.

---

La création d'un réseau privé dédié permet aux deux pare-feux d'échanger leurs informations internes ( leurs tables d'états ).

1. **VMware :** Ajout d'une carte réseau sur la VM connectée au segment LAN **SEG-SYNC**.

## 🧠 Étape 4 : Configuration du CARP (IP Virtuelles)

Le protocole **CARP** (Common Address Redundancy Protocol) permet à nos deux pare-feux de partager une adresse IP commune (la VIP). C'est cette adresse "virtuelle" qui servira de passerelle (Gateway) aux clients du réseau.

<div class="rounded-xl border border-blue-500/20 p-5 bg-blue-500/5 my-6 flex items-start gap-4">
  <div class="text-2xl">💡</div>
  <div>
    <h4 class="font-semibold text-blue-500 text-sm mb-1">Le rôle de l'AdvSkew</h4>
    <p class="text-s leading-relaxed opacity-90">
      L'<strong>AdvSkew</strong> définit la priorité : sur le Maître, je le laisse à <strong>0</strong> pour qu'il s'annonce immédiatement. Sur le second, le backup, je le monte à <strong>100</strong> pour qu'il attende avant de prendre la main.
    </p>
  </div>
</div>

Sur le premier nœud, je définis les adresses virtuelles pour le LAN et le WAN. Elles doivent être dans le même sous-réseau que les IP physiques.

## 🔄 Étape 5 : Sécurisation du lien SYNC (Règles Firewall)

C'est une étape cruciale souvent oubliée : par défaut, OPNsense bloque tout trafic entrant sur une nouvelle interface. Si on ne crée pas de règles sur le segment **CLUSTERSYNC**, le Maître et l'Esclave seront incapables de se parler, rendant la synchronisation impossible.

### Pourquoi ces règles sont vitales ?
Pour qu'un cluster soit fonctionnel et transparent, trois types de flux doivent circuler librement sur le lien dédié :
1. **ICMP (Ping) :** Pour vérifier que le nœud d'en face est vivant.
2. **XMLRPC (HTTPS) :** Pour que le Maître "pousse" sa configuration (règles, alias, NAT) vers l'Esclave.
3. **PFSYNC :** Pour synchroniser les tables d'états (sessions en cours).

<div class="flex justify-center my-8">
  <div class="w-full max-w-xl">
    <img src="/images/projets/firewalling-edge/opnsense/cluster/règles-1.png" alt="Règles Master" class="rounded border border-white/10 shadow-sm" />
    <p class="text-xs opacity-50 mt-2 italic text-center">Règles sur le Maître (OPN-01)</p>
  </div>
</div>

<div class="rounded-xl border border-red-500/20 p-5 bg-red-500/5 my-6 flex items-start gap-4">
  <div class="text-2xl">⚠️</div>
  <div>
    <h4 class="font-semibold text-red-500 text-sm mb-1">Le protocole pfsync</h4>
    <p class="text-xs leading-relaxed opacity-90">
      Le <strong>pfsync</strong> est le protocole IP 240. Sans lui, le basculement ne sera pas "Stateful". Si le Master tombe, toutes les connexions TCP actives (sessions SSH, téléchargements) seront coupées car l'Esclave n'aura aucune connaissance des sessions établies par le Master.
    </p>
  </div>
</div>

<div class="rounded-xl border border-white/10 p-4 bg-white/5 my-6 text-sm">
  <strong>📝 Configuration appliquée :</strong>
  <ul class="list-disc list-inside mt-2 opacity-80">
    <li><strong>Pass ICMP :</strong> Source <code>CLUSTERSYNC net</code> vers <code>CLUSTERSYNC net</code>.</li>
    <li><strong>Pass XMLRPC :</strong> TCP sur le port d'administration (443) entre les deux IP physiques du segment SYNC.</li>
    <li><strong>Pass PFSYNC :</strong> Protocole <code>pfsync</code> autorisé sur l'interface CLUSTERSYNC.</li>
  </ul>
</div>

## 🔄 Étape 6 : Synchronisation XMLRPC

Pour éviter de configurer chaque règle deux fois, j'active la réplication automatique. Le nœud Maître va "pousser" sa configuration vers l'Esclave via le lien **SYNC** dédié.

<div class="rounded-xl border border-green-500/20 p-5 bg-green-500/5 my-6 flex items-start gap-4">
  <div>
    <h4 class="font-semibold text-green-500 text-sm mb-1">Automatisation</h4>
    <p class="text-xs leading-relaxed opacity-90">
      Grâce au protocole <strong>XMLRPC</strong>, toute modification (règles de firewall, alias, NAT) effectuée sur le Maître est instantanément répercutée sur l'Esclave.
    </p>
  </div>
</div>

### 6.1. Paramétrage du flux de synchronisation
