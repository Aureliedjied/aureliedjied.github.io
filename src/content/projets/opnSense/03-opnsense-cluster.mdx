Un pare-feu unique est un point de défaillance critique (SPOF). 
Dans cet article, nous préparons l'architecture <strong>Haute Disponibilité</strong> en ajoutant un lien de synchronisation dédié et en déployant un second nœud identique.

## 🛡️ Le concept de Failover (Basculement)

Le **Failover** garantit la continuité de service. Si le nœud maître (`OPNsense-01`) tombe, le nœud esclave (`OPNsense-02`) prend le relais de manière transparente. Pour y parvenir, les deux machines doivent partager leurs tables d'états via un lien physique dédié.

---

## 🔌 Étape 1 : Création du segment SYNC 

Pour qu’un cluster fonctionne, les deux pare-feux doivent pouvoir se parler en permanence.

On va donc leur créer un petit réseau privé dédié, uniquement utilisé pour qu’ils échangent leurs informations internes ( leurs tables d'états ).

1. **VMware :** Ajout d'une carte réseau sur la VM connectée au segment LAN **SEG-SYNC**.
2. **Assignation :** L'interface `em2` est détectée mais non assignée.

## 🧠 Étape 4 : Configuration du CARP (IP Virtuelles)

Le protocole **CARP** (Common Address Redundancy Protocol) permet à nos deux pare-feux de partager une adresse IP commune (la VIP). C'est cette adresse "virtuelle" qui servira de passerelle (Gateway) aux clients du réseau.

<div class="rounded-xl border border-blue-500/20 p-5 bg-blue-500/5 my-6 flex items-start gap-4">
  <div class="text-2xl">💡</div>
  <div>
    <h4 class="font-semibold text-blue-500 text-sm mb-1">Le rôle de l'AdvSkew</h4>
    <p class="text-s leading-relaxed opacity-90">
      L'<strong>AdvSkew</strong> définit la priorité : sur le Maître, on le laisse à <strong>0 ou 1</strong> pour qu'il s'annonce immédiatement. Sur le second, le backup, on le monte à <strong>100</strong> pour qu'il attende avant de prendre la main.
    </p>
  </div>
</div>

### 4.1. Configuration sur le Maître (OPNsense-01)

Sur le premier nœud, nous créons les adresses virtuelles pour le LAN et le WAN. Elles doivent être dans le même sous-réseau que les IP physiques.

  <div class="w-full max-w-xl">

    />
    <p class="text-xs opacity-50 mt-2 italic text-center">Configuration des VIP : VHID 1 (LAN) et VHID 2 (WAN) avec AdvSkew à 0.</p>
  </div>
</div>

### 4.2. État sur le Secondaire (OPNsense-02)

Grâce à la synchronisation XMLRPC (étape suivante), les adresses IP virtuelles apparaissent automatiquement sur le deuxième pare-feu, mais avec un statut différent.

    />
    <p class="text-xs opacity-50 mt-2 italic text-center">Statut CARP : Le nœud secondaire est bien en mode BACKUP grâce au Skew de 100.</p>

## 🔄 Étape 5 : Synchronisation XMLRPC

Pour éviter de configurer chaque règle deux fois, nous activons la réplication automatique. Le nœud Maître va "pousser" sa configuration vers l'Esclave via le lien **SYNC** dédié.

<div class="rounded-xl border border-green-500/20 p-5 bg-green-500/5 my-6 flex items-start gap-4">
  <div>
    <h4 class="font-semibold text-green-500 text-sm mb-1">Automatisation</h4>
    <p class="text-xs leading-relaxed opacity-90">
      Grâce au protocole <strong>XMLRPC</strong>, toute modification (règles de firewall, alias, NAT) effectuée sur le Maître est instantanément répercutée sur l'Esclave.
    </p>
  </div>
</div>

### 5.1. Paramétrage du flux de synchronisation

Sur **OPNsense-01**, rendez-vous dans `Système > Haute-disponibilité > Paramètres`. Nous devons cibler l'IP physique du second pare-feu sur le segment privé.

<div class="flex justify-center my-8">
  <div class="w-full max-w-xl">
    <img
      src="/images/projets/firewalling-edge/opnsense/cluster/16.png"
      alt="Configuration du serveur distant pour la synchro XMLRPC"
      class="rounded-lg border border-white/10 shadow-sm w-full object-contain"
    />
    <p class="text-xs opacity-50 mt-2 italic text-center">Cible : 172.16.0.2 (IP de l'interface SYNC d'OPN-02).</p>
  </div>
</div>

### 5.2. Sélection des services à synchroniser

Il est crucial de cocher les éléments que l'on souhaite dupliquer. Pour un cluster standard, on sélectionne au minimum : *Firewall Rules, NAT, Aliases et Virtual IPs*.
