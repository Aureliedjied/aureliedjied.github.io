Par défaut, certaines requêtes LDAP peuvent circuler en clair sur le réseau via le port **389**, notamment lors d’un *simple bind*.  
LDAP peut toutefois être sécurisé soit par **StartTLS**, soit par **signature LDAP**.

Dans ce lab, nous choisissons d’implémenter **LDAPS (port 636)** afin de disposer d’un canal chiffré explicite et facilement vérifiable.

> 🎯 Objectif du lab  
> Ce projet s’inscrit dans une démarche de **durcissement volontaire** du socle Active Directory.  
> Certaines configurations appliquées ici seraient précédées d’une phase d’audit et de compatibilité en production.

Les risques d’un LDAP non sécurisé incluent :

- l’exposition d’identifiants  
- la divulgation d’informations d’annuaire  
- l’interception ou la modification de requêtes d’authentification  

Dans une infrastructure durcie suivant les recommandations de l’**ANSSI**, ces échanges doivent impérativement être protégés.

Le passage au **LDAPS (LDAP over TLS – port 636)** permet d’assurer la confidentialité des communications.  
Cette sécurisation repose directement sur :

- la **PKI interne** déployée précédemment  
- la **cohérence DNS**, nécessaire à la validation du certificat du contrôleur de domaine  

LDAPS nécessite la présence d’au moins un certificat valide contenant l’usage **Authentification du serveur** dans le magasin ordinateur du contrôleur de domaine.  
Dans ce lab, nous gérons explicitement ce certificat afin de maîtriser le fonctionnement du service.


## 🛠️ Étape 1 : Création du modèle de certificat "Lab-DC-LDAPS"

Dans un environnement réel, Active Directory fonctionne déjà sans LDAPS obligatoire.  
Ici, nous forçons volontairement la configuration afin d’observer les mécanismes de sécurisation des flux d’annuaire.

Pour que LDAPS soit activé, le contrôleur de domaine doit disposer d’un certificat contenant l’usage **Authentification du serveur**.

Sur l’autorité de certification, j’ouvre la console **`certtmpl.msc`** et je duplique le modèle **Authentification Kerberos** afin de créer un modèle dédié et maîtrisé.

## 📜 Étape 2 : Publication du modèle sur la CA

Une fois le modèle configuré, il doit être publié pour être utilisable.

Dans **`certsrv.msc`** :

1. Clic droit sur **Modèles de certificats**
2. **Nouveau → Modèle de certificat à délivrer**
3. Sélection du modèle **Lab-DC-LDAPS**

Le modèle est désormais disponible pour l’enrôlement par les contrôleurs de domaine.

## 🖥️ Étape 3 : Inscription manuelle du certificat sur le DC

Pour garder un contrôle précis dans ce lab, j’effectue une **inscription manuelle** plutôt qu’un auto-enrôlement via GPO.

### 🔧 Ajout du composant Certificats

1. Lancer `mmc.exe`  
2. **Fichier → Ajouter/Supprimer un composant logiciel enfichable**  
3. Sélectionner **Certificats**  
4. Choisir **Compte d’ordinateur**  
5. Sélectionner **Ordinateur local**

## 🧪 Étape 5 : Test fonctionnel avec LDP.exe

Pour valider le chiffrement réel du flux :

1. Lancer `ldp.exe`
2. **Connexion → Se connecter**
3. Serveur : FQDN du DC  
4. Port : **636**  
5. Cocher **SSL**

L’affichage du **RootDSE** confirme que la session LDAP est établie via TLS.
